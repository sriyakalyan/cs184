Overview

In this project, I built a physically-based cloth simulator and real-time renderer in 5 parts.

  1. Masses and Springs : I discretized a 2D sheet into a grid of point masses and connected them using structural, shearing, and bending springs.
  2. Simulation via numerical integration : I implemented a solver using Verlet integration to update positions based on gravity and spring forces, and applied Provot's 10% elongation constraints to ensure physical stability.
  3. Handling collisions with other objects : I built collision logic for interacting with external objects, allowing the cloth to realistically drape over spheres and rest flatly on planes without clipping.
  4. Handling self-collisions : I implemented spatial hashing to map 3D volumes to a hash table, keeping cloth folding and crumpling into itself from \(O(n^2)\) to \(O(n)\) complexity.
  5. Shaders : I moved the rendering process to the GPU, implementing several shading models including Blinn-Phong, texture mapping, environment mapped reflections, and high-detail displacement mapping.

Part 3: Handling collisions with other objects

1. Implementation Overview

Sphere Collision: I implemented Sphere::collide by calculating the distance between the point mass and the sphere's origin. If the distance is less than the radius, the point is "inside." I then project the point to the surface (tangent point) and calculate a correction vector from the point's last_position to this tangent point. The new position is the last_position plus this correction, scaled by (1 - friction).

Plane Collision: In Plane::collide, I used the dot product of the point's position relative to the plane's normal to detect a crossover. If the sign of the dot product changes between last_position and position, the mass has punched through. I calculate the intersection point and bump the mass back to the original side using a small SURFACE_OFFSET, again accounting for friction.

Part 4: Handling self-collisions

1. Implementation Overview

To implement efficient self-collisions, I used spatial hashing to avoid the \(O(n^2)\) cost of checking every point mass pair. The process involves three steps:

  • hash_position: Discretizes 3D space into boxes of size \(w \times h \times t\). Each box is mapped to a unique float key using a polynomial hash with large prime numbers to minimize collisions.
  • build_spatial_map: Clears the previous frame's map and repopulates it by hashing every point mass position and storing its pointer in the corresponding bucket.
  • self_collide: For a given mass, it retrieves all candidates from its specific hash bucket. If any candidate is within \(2 \times thickness\), a correction vector is calculated to push them apart. These corrections are averaged and scaled by simulation_steps to maintain stability.

Part 5: Shaders

1. What is a Shader Program?

A shader program is code that runs in parallel on the GPU's hardware. Thousands of instances are processed at the same time and there are two steps to the shader.

  • Vertex Shaders: These process each of our vertices in the mesh. They are responsible for transforming 3D positions into 2D coordinates and calculating vertex data like normals and tangents.
  • Fragment Shaders: After rasterization, the GPU determines which pixels a triangle covers. The fragment shader calculates the final color of each pixel using data passed down from the vertex shader.

Together, they create materials by using the vertex shader to define the geometry and orientation and the fragment shader to calculate light interaction based on that orientation.

2. Blinn-Phong Shading Model

The Blinn-Phong model is an approximation of how light reflects off surfaces. It breaks lighting into three parts:

  • Ambient: a constant low-level light applied to every part of the object so that shadows are not perfectly black, simulating indirect light bouncing around a room
  • Diffuse: based on Lambert's Cosine Law; calculates light scattering equally in all directions; brightness depends on the angle between the surface normal and the light source
  • Specular: creates the "shiny" highlights using the halfway vector to determine if the camera is in the perfect spot to see a bright reflection

4. Bump vs. Displacement Mapping

Bump Mapping is a shading trick. It uses a height map to tilt the surface normals in the fragment shader, creating shadows that look like bumps. However, the vertices never move, meaning the edges of the object remain perfectly smooth.

Displacement Mapping is a geometric change. It uses the height map in the vertex shader to physically move vertices along their normals. This creates real ridges and valleys that can be seen on the object's silhouette.

Mesh Coarseness Comparison (Displacement Mapping)

When using -o 16 -a 16, the displacement looks very spiky and jagged. This is because there aren't enough vertices to represent the curves of the texture, so each vertex becomes a sharp point. When increased to -o 128 -a 128, the "spikes" blend together into a high-fidelity, rocky surface.
